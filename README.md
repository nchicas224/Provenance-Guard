# Provenance Guard

Provenance Guard is an advisory attribution backend for creative text submissions. It does not prove whether a work was written by a human or generated by AI. Instead, it combines semantic and stylometric signals, represents uncertainty explicitly, logs decisions for auditability, and provides an appeals path for creators.

The v1 project is a Flask MVP focused on text submissions, API gateway design, data pipeline design, audit logging, AI safety guardrails, and responsible AI system evaluation.

## API Gateway

The API Gateway in this project is the combined boundary of:

- `provenance_guard/app.py`
- route blueprints in `provenance_guard/routes/`
- `Flask-Limiter`
- `RequestValidator`
- `SubmitService`
- `ContentTypeRouter`

The gateway receives public HTTP requests, applies rate limiting, validates request envelopes, creates request logs, and routes supported content types. The gateway hands off to the content pipeline at:

```text
ContentTypeRouter -> TextPipeline
```

## Setup

Create and activate a virtual environment, then install dependencies:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

Create a local `.env` file with:

```text
GROQ_API_KEY=your_key_here
```

Run the app:

```powershell
python run.py
```

Run the optional Gradio evaluation UI in a second terminal:

```powershell
python gradio_app.py
```

By default, the Gradio UI calls the Flask API at `http://127.0.0.1:5000`. To use a different API URL, set `PROVENANCE_API_BASE_URL`.

Run tests:

```powershell
python -m pytest
```

## Endpoints

### `POST /api/v1/submit`

Submits text for advisory attribution analysis.

Request:

```json
{
  "creator_id": "user_123",
  "content_type": "text",
  "content": "Submitted creative text goes here.",
  "metadata": {
    "platform": "writing_platform",
    "submission_id": "post_456",
    "title": "Optional title"
  }
}
```

Response:

```json
{
  "audit_id": "audit_abc123",
  "creator_id": "user_123",
  "content_type": "text",
  "attribution_result": "uncertain",
  "ai_likelihood": 0.58,
  "confidence_score": 0.32,
  "confidence_level": "low",
  "transparency_label": "This submission could not be confidently attributed...",
  "appeal_guidance": null,
  "signals": [],
  "degraded": false
}
```

### `POST /api/v1/appeals`

Creates an `under_review` appeal linked to an original attribution decision.

Request:

```json
{
  "audit_id": "audit_abc123",
  "creator_id": "user_123",
  "reason": "This is my original work and I can provide drafts.",
  "contact": {
    "email": "creator@example.com"
  }
}
```

Response:

```json
{
  "appeal_id": "appeal_789",
  "audit_id": "audit_abc123",
  "creator_id": "user_123",
  "status": "under_review",
  "created_at": "2026-06-29T14:30:00Z"
}
```

### `GET /api/v1/health`

Checks app and SQLite reachability. This endpoint does not call Groq or run classification signals.

```json
{
  "status": "ok",
  "service": "provenance-guard",
  "version": "v1",
  "database": "reachable"
}
```

## Detection Signals

### Groq Semantic Signal

The Groq signal uses `llama-3.3-70b-versatile` to evaluate semantic, rhetorical, and presentation-level patterns.

It measures:

- generic or broad phrasing
- repetitive structure
- overly polished transitions
- lack of concrete specificity
- template-like semantic patterns

Blind spots:

- formal, edited, academic, translated, neurodivergent, or non-native English writing can appear AI-like
- AI-generated text can include personal detail or varied structure
- model output may be sensitive to prompt wording
- this signal does not prove authorship

### Stylometric Signal

The stylometric signal uses deterministic Python heuristics.

It measures:

- vocabulary diversity
- sentence length variance
- punctuation density
- average sentence complexity
- complexity variance

Blind spots:

- short text produces unstable metrics
- genre and writing purpose change structure
- polished human writing can look uniform
- AI writing can be prompted to vary structure

## Confidence and Results

The API returns both `ai_likelihood` and `confidence_score`.

- `ai_likelihood`: direction of evidence, where `1.0` is more AI-like and `0.0` is more human-like.
- `confidence_score`: how strongly the system trusts the final attribution result.

Attribution thresholds:

| AI likelihood | Result |
| --- | --- |
| `0.00 - 0.35` | `likely_human` |
| `0.36 - 0.69` | `uncertain` |
| `0.70 - 1.00` | `likely_ai` |

Confidence labels:

| Confidence score | Label |
| --- | --- |
| `0.00 - 0.39` | `low` |
| `0.40 - 0.74` | `medium` |
| `0.75 - 1.00` | `high` |

The uncertainty band is intentionally wide because false positives against human creators are the highest-risk failure mode.

## Transparency Labels

| Attribution result | Confidence level | Label text |
| --- | --- | --- |
| `likely_ai` | `high` | "This submission shows strong patterns that are consistent with AI-generated text. This label is based on multiple signals, but it should still be reviewed with context before any action is taken." |
| `likely_ai` | `medium` | "This submission shows some patterns that are consistent with AI-generated text. The result is not definitive and should be reviewed with caution." |
| `likely_human` | `high` | "This submission shows strong patterns that are consistent with human-written text. This label is based on multiple signals, but it should not be treated as absolute proof of authorship." |
| `likely_human` | `medium` | "This submission shows some patterns that are consistent with human-written text. The result is not definitive and should be reviewed with caution." |
| `uncertain` | `low` | "This submission could not be confidently attributed. The available signals are mixed or insufficient, so this result should be treated as inconclusive." |

Degraded analysis add-ons include:

- "Semantic analysis was unavailable, so this result relies more heavily on stylometric signals and has reduced confidence."
- "Stylometric analysis was unavailable, so this result relies more heavily on semantic analysis and has reduced confidence."
- "The attribution signals were unavailable, so the system could not make a reliable classification."
- "The submission is short, so there may not be enough evidence for a reliable attribution."
- "The analysis signals did not fully agree, so this result has reduced confidence."

## Submission Size Limit

For v1, text submissions are limited to `8,000` characters. This is below the Groq model's technical context limit by design. The limit keeps the MVP focused on short-form creative submissions, protects the free-tier API budget, reduces latency, keeps audit entries readable, and makes abusive bulk submissions easier to reject.

## Rate Limiting

The submit endpoint is limited to:

```text
10 submissions per minute per client
100 submissions per day per client
```

This threshold is intended to fit short-form creative platform behavior. A creator may test a few drafts or retry after edits, but a normal user is unlikely to submit more than 10 attribution requests in one minute. Flood attacks, broken scripts, and accidental loops should exceed that threshold quickly.

Rate-limited response:

```json
{
  "error": {
    "code": "rate_limited",
    "message": "Too many submissions. Please wait before trying again."
  }
}
```

v1 rate limiting is educational and not production-grade authentication. A production system should use authenticated identity, API keys, gateway enforcement, and persistent rate-limit storage.

## Evaluation Report

The goal of evaluation is not to prove detector accuracy. The goal is to demonstrate system behavior, uncertainty handling, degraded-mode behavior, auditability, and false-positive caution.

Minimum evaluation set:

| Scenario | Expected safety behavior | Result fields to record |
| --- | --- | --- |
| Human-written informal paragraph | Should avoid overclaiming AI. | result, ai_likelihood, confidence, label, audit_id |
| Human-written formal paragraph | Should be cautious because formal writing may look AI-like. | result, ai_likelihood, confidence, label, audit_id |
| AI-generated generic paragraph | May return likely_ai if signals align. | result, ai_likelihood, confidence, label, audit_id |
| AI-generated personal-style paragraph | Should not assume personal detail proves human authorship. | result, ai_likelihood, confidence, label, audit_id |
| Very short text | Should return uncertain or low confidence. | result, ai_likelihood, confidence, label, audit_id |
| List-like text | Should reduce confidence because structure is unusual. | result, ai_likelihood, confidence, label, audit_id |
| Groq unavailable | Should degrade gracefully and cap confidence. | result, degraded flag, label, audit_id |
| Appeal after likely_ai | Should create under_review appeal linked to creator_id and audit_id. | appeal_id, status, audit_id |

The manual evaluation fixture lives at `docs/evaluation_examples.json`. It
contains 12 examples across expected `likely_human`, `likely_ai`, and
`uncertain` buckets. The Gradio UI loads these examples into the Submit tab so
each case can be run through the Flask API and recorded in this report with the
actual `attribution_result`, `ai_likelihood`, `confidence_score`,
`confidence_level`, `transparency_label`, and `audit_id`.

The test suite covers request validation, stylometric scoring, confidence scoring, submit API behavior, health checks, and appeals. Groq is mocked in tests so automated results do not depend on network access, free-tier rate limits, or model output drift.

## Safety Boundaries

- The system is advisory.
- Authorship cannot be proven from text alone.
- `likely_ai` is not an accusation of misconduct.
- Low-confidence non-uncertain results collapse to `uncertain`.
- Appeals are accepted into `under_review`; final human review is outside v1 scope.
- Final attribution decisions should not be returned unless audit logging succeeds.
